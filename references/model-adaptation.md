**DeepSeek 专项优化（当前模型）**：

```
Phase 1 推理增强：
  - 启用 thinking 模式：在画蓝图前，先思考 "哪些模块划分方式会让依赖最少？"
  - 作用：thinking tokens 免费且不占最终输出，适合架构推演

Phase 2 前缀缓存利用：
  - BLUEPRINT.md 修改时追加段落而非替换原有段落
  - @MODULE 接口签名在每次 [done] 后追加完整签名（含 pre/post/error/side-effect）
  - 效果：缓存命中率可从 ~60% 提升到 ~90%，每次填充成本降低 ~10×

Phase 3 深度验证：
  - 运行时验证前，用 thinking 预判每条 @FLOW 最可能在哪步出错
  - 优先测试预判的高风险路径
```

**模型自检**：执行前确认 "我是什么模型" → 查表取对应参数列 → 按该列执行。

### 新模型自动适配

未来模型未在表中时，按以下能力维度自分类，无需更新 skill：

```
问自己 4 个问题 → 得到 4 个策略参数：

1. 上下文窗口多大？
   ≥ 500K  → 始终全量加载蓝图，释放阈值 ~65%
   200K    → Phase 1 全量，Phase 2 按需加载段落，阈值 ~50%
   128K    → 仅加载当前模块+依赖签名，阈值 ~40%

2. 是否支持 thinking / extended reasoning？
   是 → Phase 1 深度推理（探索 2-3 种方案）
         Phase 3 预判高风险路径
   否 → Phase 1 标准推理（1 种方案）

3. 代码生成质量如何？（自评：高/中/需验证）
   高   → Phase 2 抽样对照，信任实现
   中   → Phase 2 全量逐行对照
   需验证 → Phase 2 全量 + Phase 3 完整运行时验证

4. 是否支持子任务并行？
   是 → 同层 ≤ 3 并行填充
   否 → 串行填充
```

**示例**：一个 2027 年发布的未知模型，512K 上下文 + thinking + 自评代码质量"中"+ 支持并行 →
- 全量蓝图、阈值 65%、深度推理、全量对照、≤ 3 并行

**模型自检**：执行前确认 "我是什么模型" → 优先查表；不在表中则运行以上 4 个问题自分类。

### 能力探针（每次会话首次触发时执行）

skill 激活后立即运行 5 秒自测，验证适配参数是否与实际环境匹配：

```
探针 1 — 上下文加载验证
  操作: 加载 BLUEPRINT.md（如果存在），记录 token 消耗
  预期: 若模型 ≥ 500K 上下文 → 全量加载无报错
  异常: 若加载失败 → 降级为按需加载策略

探针 2 — 蓝图修改方式验证
  操作: 对 BLUEPRINT.md 执行一次微小修改（如追加时间戳注释）
  预期: 若模型支持 prefix cache → 使用追加模式（影响缓存命中率）
  异常: 若追加失败 → 回退为替换模式，标记缓存策略为"标准"

探针 3 — 并行能力验证
  操作: 如有 ≥ 2 个同层 [empty] 模块，尝试并行启动子任务
  预期: 若平台支持 → 子任务成功创建
  异常: 若子任务创建失败 → 标记并行策略为"串行"

探针 4 — thinking 模式检测
  操作: 尝试输出一段 thinking token
  预期: 若模型支持 → thinking 成功渲染（静默，不占输出）
  异常: 若 thinking 被截断或转为普通文本 → 标记为"无 thinking"，Phase 1 降为标准推理
```

探针结果记录在 CHECKPOINT.md 的 `@PROBE` 段：
```
@PROBE 2026-05-10 DeepSeek V4
  上下文: 全量加载 ✅
  缓存: 追加模式 ✅
  并行: 支持 ✅ (≤3)
  thinking: 支持 ✅
```

任一探针失败 → 自动回退到对应降级策略，不阻塞后续流程。
